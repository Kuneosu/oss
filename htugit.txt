mkdir oss  : oss 라는 폴더 생성

cd oss : oss 폴더로 이동

git clone https://github.com/Kuneosu/oss.git .  : 현재 폴더에 oss.git의 클론 저장소 생성

git remote -v  :  git 과 연결된 github 저장소 확인

git log  :  git 의 로그 확인

code index.html  :  vscode로 index.html 파일 열기

git status  :  git 의 상태 확인

git add index.html  : 수정한 index.html 파일을 git 에 추가

git config --global user.email  "kks92155784@gmail.com"
git config --global user.name "Kuneosu"  : git 사용자의 이름과 이메일 설정

git commit -m "git 내용수정"  변경된 내용의 커밋메시지 작성 

git push -u origin master  :  변경된 파일과 커밋을 github로 전송

git pull  :  다른 컴퓨터에서 변경된 github를 원본 컴퓨터에서 다운로드 ( 원본파일이 삭제되어 해보지 못함 )

브랜치 : 나뭇가지 느낌, 

master : C1(커밋1) - C2 - C3 (여기서 test 브랜치 생성)
		test  : C4 ( test 브랜치에서 생성한 커밋4)
이 경우 마스터브랜치에서는 C4의 변경내용이 반영되어 있지 않음 ( 작업의 분리가 가능 )
여러 브랜치를 오가며 복원 포인트를 잡아두고 다양한 방향으로 작업을 진행해볼 수 있음
그러나 master 브랜치를 복원 포인트로 사용하는 것은 아닌듯
여러 브랜치를 point1(긍정적방법), point2(부정적방법)  과 같은 식으로 브랜치를 생성 후 
한가지 방향을 결정했다면 master(결론) 으로 merge(병합) 시켜서 핵심 가닥으로 사용하는게 좋아보임
차후 master(결론) 브랜치에서 문제가 생겼을 경우 이전에 갈라졌던 복원 브랜치로 돌아가서
재작업을 해보는 방법을 통해 문제의 수정에 용이할 것이라 예상됨.

git branch test  :  test 라는 이름의 브랜치 생성

git branch -v  :  생성되어있는+선택되어있는 브랜치 확인 

git checkout test  :  test 라는 이름의 브랜치 선택

cat index.html  :  index.html 파일의 내용 확인

git merge test  :  현재 브랜치(master)로 test 브랜치의 내용을 병합 
		( 이 경우 test 브랜치에서 작업했던 내용, 커밋, 파일 등이 모두 가져와짐 )
