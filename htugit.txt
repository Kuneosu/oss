브랜치 : 나뭇가지 느낌, 

master : C1(커밋1) - C2 - C3 (여기서 test 브랜치 생성)
		test  : C4 ( test 브랜치에서 생성한 커밋4)
이 경우 마스터브랜치에서는 C4의 변경내용이 반영되어 있지 않음 ( 작업의 분리가 가능 )
여러 브랜치를 오가며 복원 포인트를 잡아두고 다양한 방향으로 작업을 진행해볼 수 있음
그러나 master 브랜치를 복원 포인트로 사용하는 것은 아닌듯
여러 브랜치를 point1(긍정적방법), point2(부정적방법)  과 같은 식으로 브랜치를 생성 후 
한가지 방향을 결정했다면 master(결론) 으로 merge(병합) 시켜서 핵심 가닥으로 사용하는게 좋아보임
차후 master(결론) 브랜치에서 문제가 생겼을 경우 이전에 갈라졌던 복원 브랜치로 돌아가서
재작업을 해보는 방법을 통해 문제의 수정에 용이할 것이라 예상됨.

git branch test  :  test 라는 이름의 브랜치 생성

git branch -v  :  생성되어있는+선택되어있는 브랜치 확인 

git checkout test  :  test 라는 이름의 브랜치 선택

cat index.html  :  index.html 파일의 내용 확인

git merge test  :  현재 브랜치(master)로 test 브랜치의 내용을 병합 
		( 이 경우 test 브랜치에서 작업했던 내용, 커밋, 파일 등이 모두 가져와짐 )

reset 과 revert 
: 리셋은 커밋을 삭제하고 이전 커밋으로 돌아가는 동작이고
리버트는 커밋을 삭제하지 않고 이전 커밋으로 돌아가는 새로운 커밋을 작성하여
이전 커밋으로 돌아가는 동작이다.

리셋을 할경우

GITHUB상의 커밋 : C1-C2-C3-C4
컴퓨터1의 로컬저장소 : C1-C2-C3-C4
컴퓨터2의 로컬저장소 : C1-C2-C3-C4

이렇게 두 명의 사람이 동시에 같은 GITHUB의 커밋들을 복제하여
작업을 진행중이라고 할 때 컴퓨터 1에서 C4를 reset 하여
C3로 돌아가버리고 push를 하게되면 

GITHUB상의 커밋 : C1-C2-C3
컴퓨터1의 로컬저장소 : C1-C2-C3
컴퓨터2의 로컬저장소 : C1-C2-C3-C4

위와 같은 커밋이 형성되고 이 상태에서 
컴퓨터2에서 새로운 작업 내용을 C5로 추가하여 
깃허브에 push 하게 되면 커밋의 수가 다르므로 오류가 발생하게 된다.
따라서 여러 로컬저장소에서 공개github를 통해 작업하고 있을 경우
reset은 절대 해선 안된다.

그 경우 사용하는것이 revert 이다. 

C4에서 C3로 돌아가는 행위를 C5로 새로운 커밋을 작성하여 업데이트 하기 때문에
다른 로컬저장소에서 push를 하여도 문제가 되지 않는다.


git reset HEAD^  :  HEAD가 가리키고 있는 커밋의 한 단계 앞으로 돌아갑니다.
		 이 경우 HEAD가 가리키고 있던 커밋은 삭제되지만 파일의 작업내용은 삭제되지 않습니다.
		 ( soft, mixde reset : 커밋만 삭제 / hard reset : 작업내용까지 삭제 )
		또한 두 단계 앞으로 돌아갈려면 ^ 표시를 하나 추가하면 됩니다.

git revert HEAD  :  revert의 경우 HEAD가 가리키고 있던 커밋을 없앤 커밋을
		다음 커밋에 추가하는 것이기 때문에 ^ 표시가 필요하지 않다.
		또한 revert를 진행하면 파일의 작업내용도 커밋과 함께 사라지게된다.


요약 : reset 은 혼자서 작업을 진행중일 때 사용하고
	revert 는 하나의 github를 두고 협업을 진행중일때 사용하는 것이 조으다.
